1. ACCOUNTS & AUTH
────────────────────────────────────────────────────────────────────────────────

Registration
  - Username: 3–20 chars, letters/numbers/underscore
  - Password: 4–100 chars
  - Email: optional, used only for password reset
  - On register: session token is issued immediately (auto-login)

Login
  - Username + password
  - Password hashed with PBKDF2-SHA256, 100,000 iterations, random salt (16 bytes)
  - Returns a session token stored in localStorage

Logout
  - Deletes session from DB + in-memory cache

Password Reset
  - User submits email → server sends reset link via Resend API (or SMTP fallback)
  - Token expires after 1 hour
  - On reset: all existing sessions are invalidated

Session system
  - Tokens stored in `sessions` table (token → player_id)
  - In-memory session cache (10-minute TTL) avoids DB round-trips on every request
  - In-memory player data cache (1-hour TTL) — zero DB reads on authenticated hot path

2. SAVE / LOAD SYSTEM
────────────────────────────────────────────────────────────────────────────────

How saving works
  - Client calls POST /api/save with full state JSON every 30 seconds
  - Also saves immediately after any purchase (weapon, skill, area unlock)
  - On tab close: navigator.sendBeacon → POST /api/save (with token in body since
    sendBeacon can't set headers); XHR sync fallback
  - Server saves synchronously with 3 retries before returning HTTP 200
    (ensures progress is durable — no fire-and-forget that could be lost on restart)

State versioning
  - Every player state has a `stateVersion` field (currently v7)
  - _upgrade_state() migrates old saves to current version
  - v1: baseline keys
  - v2: totalClicks
  - v3: inventory + potion timers
  - v4: weapon index remapped (5 → 15 weapons)
  - v5: area progression (currentArea, unlockedAreas)
  - v6: general skill system replaced (IDs 1–12 → 11–34)
  - v7: knight + sorcerer skill trees replaced (3×4 grids)

What is saved (in `state` JSON blob)
  score, gold, exp, level, weaponIndex, skillPoints (general),
  knightSkillPts, sorcSkillPts, ascended, ascendedClass,
  currentArea, unlockedAreas, inventory, all potion timers,
  bossSpawnCounter, hmmCooldownEnd, arcaneWeakeningStacks,
  suddenDeathCooldownEnd, essenceGatheringEnd, totalClicks,
  legacy fields kept for migration (firestormCharges, comboStacks, etc.)

4. COMBAT & GAME LOOP
────────────────────────────────────────────────────────────────────────────────

Click combat
  - Player clicks a monster to deal damage (rollClickDmg)
  - Base damage from current weapon (min–max range)
  - 0.35s manual click cooldown (BASIC_COOLDOWN_MS)
  - Knight K1–K4, K5 modify click damage
  - clicking cooldown should be influenced by cooldown reduction potions and other general cooldown reduction mechanisms

Auto-attack (A1 skill required)
  - Fires every effectiveAutoCooldown() = max(200ms, 700ms – A1pts×50ms)
  - Potion of Swiftness also reduces all cooldowns by 10/15/25%
  - Targets lowest-HP worm first
  - A2 skill: +10%/pt auto damage
  - A3: 10%/pt chance of hitting a 2nd target
  - A4: 10%/pt chance of hitting a 3rd target

Monster spawning
  - Multiple worms on screen simultaneously
  - the number of monsters is 10 at level 1 and in the first arena
  - every new arena unlocks additional 1 monster (so 11 monsters in second location, 12 monsters in third location and so on)
  - every 50 player levels there will be an additional monster
  - Boss spawn after first 10 kills for the given player (to quickly show the mechanics)
  - Generally, after every monster kill there is 2% chance for boss spawn
  - Uber-boss has 1% chance to spawn (so first the game checks if the uber boss rolled, then, if the boss rolled)
  - Potion of Madness: +10 max spawn count, ×10 spawn rate
  - monster spawn rate should scale with level and locations to ensure that on high levels the player doesn't constantly run out of monsters

Level system
  - XP formula: expForLevel(x) = floor(50 × (x³ – 6x² + 17x – 12) / 3)      (Tibia formula)
  - No level cap (practical upper limit ~300 to reach The Void)
  - Level-up flash message shown for ~2s

Score
  - +1 per normal mob kill, +BOSS_KILLS (10) per boss kill

18. SERVER INFRASTRUCTURE
────────────────────────────────────────────────────────────────────────────────

Runtime
  - Python 3, stdlib only (no pip except libsql_experimental for Turso + optional email libs)
  - ThreadingHTTPServer overridden with ThreadPoolExecutor(max_workers=64)
  - SSE connections bypassed to dedicated daemon threads (not pool workers)
  - Static files preloaded into RAM at startup (in-memory dict)

Database
  - Turso (remote SQLite via libsql_experimental) when TURSO_URL + TURSO_AUTH_TOKEN set
  - Falls back to local SQLite (game.db) for local dev
  - Single global Turso connection with auto-reconnect on stream expiry
  - Write lock for all INSERT/UPDATE/DELETE operations

Email
  - Resend HTTP API (preferred — works on Railway)
  - SMTP fallback (may be blocked on Railway)

Environment variables
  PORT, TURSO_URL, TURSO_AUTH_TOKEN, ADMIN_TOKEN,
  RESEND_API_KEY, RESEND_FROM, NOTIFY_EMAIL, GAME_URL,
  SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, SMTP_FROM,
  DATA_DIR (optional custom DB directory)

Deployment
  - Railway (production from master branch, test from dev branch)
  - Procfile: web: python server.py
  - runtime.txt: python-3.11.x
